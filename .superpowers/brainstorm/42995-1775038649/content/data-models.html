Data Models — DynamoDB Schema

Design Section 2 of 4 — 5 tables, each with a clear single purpose.

📦 projects
Attribute Type Notes
pk 🔑 String e.g. jamf-splunk-integration
title String Display name
description String Short summary
category String jamf | mcri
tech_stack List ["Flask","Lambda","Splunk"]
current_phase String Discovery→Build→Launch
phases Map Phase notes per stage
like_count Number Atomic increment
created_at String ISO timestamp
🌅 sunset_photos
Attribute Type Notes
pk 🔑 String e.g. sunset-vic-falls-01
s3_url String Full S3 image URL
location String e.g. "Victoria Falls"
story String Caption / short story
taken_at String Date photo was taken
like_count Number Atomic increment
created_at String ISO timestamp
📚 books
Attribute Type Notes
pk 🔑 String e.g. atomic-habits
title String Book title
author String Author name
status String reading | read
reflection String One-line takeaway
💬 comments
Attribute Type Notes
pk 🔑 String home, project#slug, photo#id
sk 🔑 String Timestamp (newest-first sort)
author String Visitor's name
body String Comment text
One table handles all comment types. pk is the parent item, sk is the timestamp — querying by pk returns all comments newest-first.
🌍 page_visits — one row per page, atomically incremented
Attribute Type Example
pk 🔑 String home, sunsets, projects
count Number 1,204
last_visited String ISO timestamp
Special row: pk = "global"
Holds the total across all pages — shown in the nav bar as 🌍 1,204 visits.

Each page visit triggers a DynamoDB ADD operation (atomic increment) — safe even if two people visit at exactly the same time.
Summary: 5 DynamoDB tables — projects · sunset_photos · books · comments · page_visits
Photos stored in S3 (URLs saved in DynamoDB). No user accounts needed — visitors comment with just a name.
