3 Architecture Approaches

Same stack at the core (Flask + Lambda + DynamoDB) — the difference is how the frontend interactivity is handled.

A

Flask + HTMX Recommended

Flask renders all HTML server-side (Jinja2 templates). HTMX adds interactivity by swapping small HTML fragments — likes, comments, and visit counts update without a page reload, but you write almost no JavaScript.

Pros

  • Very beginner-friendly — no JS framework
  • Comments, likes, visit counts work out of the box
  • Easy to debug (just HTML + Python)
  • Works great with Flask + Jinja2

Cons

  • One extra library to learn (HTMX — small)
  • More Lambda calls per interaction
B

Flask + Vanilla JavaScript

Flask renders pages server-side. All interactive features (likes, comments, visit counts) use fetch() calls to Flask API endpoints. You write the JS yourself — full control, no extra libraries.

Pros

  • No extra dependencies
  • Good learning opportunity for JS
  • Fine-grained animation control

Cons

  • More JS to write and maintain
  • Easy to get messy as features grow
C

Flask API + Static Frontend (S3)

Flask becomes a pure JSON API on Lambda. HTML/CSS/JS lives on S3 as a static site. The frontend calls the API for all data. Closest to a modern "decoupled" architecture.

Pros

  • Frontend and backend fully independent
  • Static site = blazing fast loads
  • Scales easily

Cons

  • Most complex to set up (two separate deployments)
  • Requires more JS knowledge
  • Harder to get right for beginners

My recommendation: A (HTMX). Your prompt specifically mentioned HTMX, it fits Flask beautifully, and it means your interactive features (likes, comments, visit counts) need almost zero JavaScript. You can still sprinkle Vanilla JS for animations (heart bounce, timeline slide-in) on top.
